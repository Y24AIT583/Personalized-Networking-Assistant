Project Overview: Personalized Networking Assistant
This Entity-Relationship (ER) diagram outlines the database schema for a Personalized Networking Assistant. The primary goal of this system is to manage corporate participation in global professional events, track individual delegate profiles, and measure engagement and networking interactions during these summits.

Core Entities & Relationships
The architecture is built around a central hub: the Networking Summit. The relationships can be broken down into three main blocks:

1. The Event & Corporate Foundation
	Corporate Profile & Networking Summit: A 1-to-Many relationship (1 -> Many). A single corporation (CorpID) can host multiple summits, but each specific summit instance is tied to one host corporation.

	Global Event Master & Networking Summit: A 1-to-Many relationship (1 -> Many). Broad global event categories (like a 'Tech' or 'Finance' master template) set the agenda for multiple individual summits.

2. The Attendee Architecture
	Delegate Profile: Represents the individuals attending. Interestingly, the diagram shows a 1-to-Many Hosts link bridging the Networking Summit to Delegate Profiles, meaning a summit hosts many delegates.

	Generates Link: A delegate profile also links downward to a secondary table (unlabeled title, but contains InsightcID), showing a 1 -> Generates relationship to log specific bio/skill variations.

3. Engagement & Analytics Tracking
	From the central Networking Summit, the system tracks three distinct streams of data using 1-to-Many relationships:

	Participation Insight: Records specific questions or engagement data generated by a delegate during a summit.

	Interaction Metric: Tracks the networking metrics (e.g., interaction types and duration in minutes) between attendees during a specific summit.

Key Technical Observations
	Primary Keys (PK): Highlighted in red/underlined text (e.g., SummitID, CorpID), ensuring unique identification for every record.

	Foreign Keys (FK): Highlighted in blue text (e.g., CorpID and GlobalEventID inside the Summit table), establishing the foundational relational links across the schema.

	Strict Business Rules: The design ensures that analytics (Participation Insight and Interaction Metric) cannot exist in a vacuum—they are rigidly anchored to both a specific SummitID and the respective delegates.
